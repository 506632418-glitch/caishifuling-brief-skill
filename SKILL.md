# 蔡氏福宁产品brief填写 v1.6.1

> **版本**：v1.6.1 | **日期**：2026-07-06

## 概述

帮助审核人将产品信息文件中的内容，结构化填写到「蔡氏福宁 · 产品brief 填写工作台」中。

**brief工具路径**：与本 SKILL.md 同目录下的 `蔡氏福宁_产品brief_填写工具.html`

**数据写入机制**：brief工具通过 localStorage 读取和存储数据。AI 维护一份 JSON 数据文件（`/tmp/brief_data.json`），每版块确认后用 `scripts/brief_inject.py` 将数据注入 HTML，页面打开时自动预填 localStorage。

**核心原则**：有据可依、无据留空、宁缺毋滥、逻辑自洽。
**防幻觉铁律**：凡不存在于信息来源中的内容，不得自行编造。
**文案基准**：参考休足贴和蒸汽眼罩两个完整brief案例的行文风格、结构模板和合规边界。
**合规底线**：蔡氏福宁产品属于日用品，所有文案不得暗示或明示医疗功效。

## ✍️ 文案风格规范

> **参考案例**：休足贴 brief、蒸汽眼罩 brief、三伏贴 brief（均位于 `/Users/fancy/Desktop/产品/产品信息整理集合/`）。填写时应参照案例的行文基调、结构组织和用词习惯。

### 语言基调

| 维度 | 要求 |
|------|------|
| 调性 | 专业可信、亲切易懂。介于"产品经理给消费者讲产品"和"朋友安利好东西"之间，不居高临下 |
| 消费者视角 | 默认"你"视角，把专业术语翻译成消费者能懂的话，用类比和具象化替代抽象描述 |
| 信息密度 | 高——每句话有信息量，不写正确的废话（如"本品品质优良"）。用数据、对比、场景支撑结论 |
| 禁止 | 浮夸营销体（"极品""神器""秒杀"）、假大空形容词堆砌、医疗用语过度包装 |

### 各版块语言风格

#### 产品口号
- 结构：4-8字短句，两到三组并列，利益导向
- 案例：`冰凉舒爽 本草解压 焕发双腿活力`（休足贴）/ `蒸汽眼部SPA 释放双眼压力`（蒸汽眼罩）/ `冬病夏治 三伏养阳`（三伏贴）
- 类型：休足贴和蒸汽眼罩是"产品体验+结果"型，三伏贴是"时令概念+功效"型——根据产品特性选择最适合的类型
- 四大手法（可组合使用）：
  1. **体验+结果**：前半段描述感官体验，后半段落到消费者获益（`冰凉舒爽 本草解压 焕发双腿活力`）
  2. **时令+功效**：绑定节气/时间概念，制造仪式感和窗口期（`冬病夏治 三伏养阳`）
  3. **场景+画面**：用一句话构建使用场景的视觉想象（如"睡前贴上 醒来轻松""八小时温热 一整晚好眠"——前半段是产品使用画面，后半段是画面延续的结果）
  4. **数字对比**：用数字制造冲击力和可信度（如"13味本草 1贴温养""30分钟蒸汽 一整天舒润"——数字让利益点从模糊变具体）
- 组合示例：`4小时温热 13味本草 一年就养这四十天`——数字+时令+画面，三重叠加
- 要求：不堆砌形容词，每个词对应一个可感知的利益点
- ⛔ 禁用夸张词："超凡""极致""无与伦比"

#### 产品口播
- 结构：痛点提问 → 产品是什么（一句话） → 核心特征 → 关键利益 → 行动号召
- 长度：150-300字
- 语气：先共情再解决（"站了一天腿又酸又胀？""每天盯屏8小时眼睛干涩？""冬天手脚冰凉，夏天吃凉就腹泻——别等到冬天才想起养阳气"）
- 节奏：短句穿插长句，念出来顺口。数字、对比、感官描述穿插其中
- 进阶手法（来自三伏贴案例）：
  - **时间窗口紧迫感**："夏天最热的三伏天……一年就这四十天"
  - **反常识对比**："别等到冬天才想起养阳气""别人抢冰镇西瓜，你翻出三伏贴"
  - **成分递进**：先讲核心差异成分（蜂蜜膏芯），再讲数量震撼（13味本草），最后讲深层机制（沿经络渗透）
- ⛔ 禁用：广播腔"本品采用…"、堆砌成分列表、假对话"你知道吗"

#### 品类定义
- 结构：`[外用/自发热 + 产品类型]——[使用部位+方式]，通过[核心机制+成分]，[核心功能]`
- 案例：`外用舒缓休足贴——贴在腿足部位，通过含温泉水的水凝胶+4味本草+清凉因子，缓解腿足不适` / `外用穴位贴敷产品——贴于穴位部位，通过蜂蜜膏芯复配13味本草对穴位持续刺激，温补阳气、通络驱寒`
- 要求：一句话说清"这东西是什么、用在哪、怎么用、能干嘛"。使用部位（腿足/眼部/穴位）是关键定位信息
- 进阶手法（来自三伏贴案例）：如果产品有使用时机限定（如三伏天），在品类定义中体现"外用 + 时效 + 类型"的结构

#### 核心技术
- 格式（每条）：`名称 — 功能 — 出处`
- 出处要求：经典文献引用原文+出处（如"伸筋草：'主久患风痹，脚膝疼冷'——《本草拾遗》"），第三方检测报告标注机构名称
- ⛔ 禁止模糊出处："科学研究表明""传统智慧认为"

#### 人群描述
- 结构：`身份标签 → 痛点 → 现有方案及不足 → 他们需要什么`
- 要求：具象到"能想象出这个人的样子"，有职业/场景/行为细节，不写"注重健康的消费者"这种泛泛标签
- 进阶手法（来自三伏贴案例）：
  - **标签化命名**：给每个人群起一个生动标签——冰块族、堵堵族、易感族、脾胃弱族，比"寒性体质人群""久坐人群"更有记忆点
  - **痛点具象到身体感受**：不只说"怕冷"，要说"夏天在空调房也要披外套""冬天手脚捂不热""吃一点凉的就腹泻"
  - **现有方案的"不够"要具体**：不是笼统说"现有方案不方便"，而是"艾灸馆预约麻烦、单次百元起步"
- 案例参考：休足贴人群1-4（职业维度：站6h+/久坐族/健身人群/暴走旅游）、三伏贴人群1-4（体质维度：冰块族/堵堵族/易感族/脾胃弱族）

#### 场景描述
- 结构：`场景画面 → 身体感受 → 使用产品 → 感受变化`
- 要求：感官细节（凉意涌上来、紧绷感松动），叙事感
- 进阶手法（来自三伏贴案例）：
  - **反常识对比开场**："入伏第一天，别人在抢冰镇西瓜，你翻出三伏贴"
  - **季节与身体的关联**：不只写使用瞬间，要把"时令/节气→身体状态→产品作用"串成一条线
  - **穴位具名**：标注具体穴位名称（大椎穴、三阴交、关元、足三里、命门、中脘、天枢），增加专业感
- 案例：休足贴场景（下班回家/健身后）、蒸汽眼罩场景（午休/睡前/差旅）、三伏贴场景（入伏首日/办公室空调/换季预防/冰饮不适）

#### POD（差异点）
- 格式：FAB三元结构（F=特征/A=优势/B=利益）+ vs竞品 + RTB证据
- 要求：F要具体（数值/成分/工艺），A要说清为什么好，B要落到消费者可感知的结果
- vs竞品：不贬低，说事实差异
- RTB：优先第三方检测报告、文献出处；其次产品实测数据

#### FAQ
- 格式：直接回答（"可以""不建议""不能"），再说原因
- 语气：朋友聊天，知道你关心什么
- ⛔ 禁用：客服体"请您放心""我们将竭诚为您服务"、回避问题兜圈子

## 🔬 缺失信息补充规则

当产品信息源文件未覆盖某字段时，AI 可基于**已确认的产品信息和参考案例风格**生成「📝AI参考方案」供审核人确认。

### 何时可以生成AI参考方案

| 场景 | 做法 |
|------|------|
| 信息源有据 | ✅ 直接引用，标注出处 |
| 信息源无据，但同类产品案例中该字段有成熟范式（如口号结构、口播结构、人群画像框架） | ☑ **可生成** AI参考方案——模仿案例的结构和语气，填入当前产品已确认的要素 |
| 信息源无据，且无法从已确认信息推断 | ⛔ 标注「信息源未提供，跳过」——**不得**凭空编造 |

### AI参考方案规则

1. **必须标注**：所有 AI 生成内容前加 `📝AI参考：` 前缀，与信息源原文明确区分
2. **事实锁**：只能基于**阶段0已确认的产品信息**（名称/成分/规格/功效/使用方式等）来生成，不得引入新的"事实"
3. **风格参考**：参照休足贴/蒸汽眼罩案例的语言基调和结构模板
4. **合规校验**：生成内容必须通过合规规则审查（见下一节）
5. **审核人确认**：AI参考方案必须经审核人确认后才可写入 HTML

### 典型适用字段

| 版块 | 可生成参考方案的字段 | 说明 |
|------|-------------------|------|
| 心智层 | 产品口号、产品口播 | 参考案例结构，用已确认的产品成分/机制/利益点填充 |
| 产品层 | 感官描述（颜色/质地/气味/触感）、包装描述 | 基于成分/品类常识推断，标注为AI参考 |
| 市场层 | 竞品优势分析 | 基于联网搜索的竞品信息，用案例中的 vs竞品 格式组织 |
| 利益层·人群 | 核心人群画像 | 基于产品定位/品类/使用场景推断人群特征 |
| 利益层·场景 | 使用场景描述 | 基于品类使用方式和已确认利益点描绘场景 |
| 常见问题 | FAQ | 基于品类常见问题+已确认产品信息生成 |

### 展示格式

```
📦 {N}/8 {版块名称}

| 字段 | 填写内容 | 信息来源 |
|------|---------|---------|
| 字段A | XXX | 产品信息卡-P2 |
| 字段B | 📝AI参考：[生成内容] | 参考休足贴/蒸汽眼罩案例风格，基于已确认的[成分/利益点]生成 |
| 字段C | ⚠️ 待确认 | 信息源未提供且无法推断 |

请确认以上内容，特别是「📝AI参考」项是否需要修改。
```

## ⚖️ 合规规范：医疗功效暗示规避

> **底线**：蔡氏福宁产品属于**日用品**，不是药品/医疗器械。所有文案不得暗示或明示医疗功效。

### 绝对禁用词

| 类别 | 禁用词（含变体） | 原因 |
|------|----------------|------|
| 治疗宣称 | 治疗、治愈、疗程、疗效、根治、祛除、药到病除 | 暗示药品功能 |
| 疾病关联 | [任何疾病名称]、消炎、抗菌、杀菌、抗病毒 | 暗示医疗作用 |
| 预防宣称 | 预防[疾病]、增强免疫力、提高抵抗力 | 暗示保健功能 |
| 诊断宣称 | 诊断、检测、筛查、判断 | 暗示医疗器械功能 |
| 绝对化 | 最、第一、独家、唯一、100%、彻底、完全、绝不 | 违反广告法 |
| 承诺性 | 保证、一定、肯定、必定、绝对有效 | 构成承诺担保 |
| 比较性贬低 | 远胜于、碾压、秒杀[竞品名]、智商税 | 不正当竞争 |

### 替代表述对照

| 禁用 | 替代（合规表述） |
|------|----------------|
| 治疗/治愈 | 舒缓、缓解不适、帮助放松（注：只能用于主观感受，不可用于疾病症状） |
| 消炎/抗菌 | 洁净、清爽、清新、洁净呵护 |
| 增强免疫力 | 养护、日常呵护、帮助维持良好状态 |
| [疾病]患者 | 特定人群/关注[某方面]的人士 |
| 保证/一定有效 | 帮助/有助/辅助 |
| 彻底解决 | 帮助缓解 |

### 合规自查话术模板

> 在填写以下字段时，必须逐句过合规：
> - 产品口播（心智层）
> - 作用原理（产品层）
> - 核心技术描述（产品层）
> - POD差异描述（核心价值层）
> - FAQ回答（常见问题层）
>
> **方法**：把每句话的"功效描述"换成"感受描述"——从"它治好了什么"变成"你感觉到了什么"。

### 合规免责声明

所有brief中涉及产品功效的表述，应在以下位置附加免责：

1. **注意事项**字段：必须包含 `本品属于日用品，不具有疾病预防、治疗功能，不可替代药品或医疗器械`
2. **FAQ**：涉及功效对比/能否治疗类问题时，开头或结尾加 `本品为日用品，非药品`

### 合规校验清单

每个版块填写完成后，AI 必须逐项自检：

```
⚖️ 合规校验
├─ 无"治疗/治愈/消炎/抗菌"等医疗宣称词
├─ 无具体疾病名称关联
├─ 无"最/第一/独家/100%"等绝对化用语
├─ 功效描述已转为"感受描述"（你感觉到了什么，而非它治好了什么）
├─ 注意事项中已包含日用品免责声明
└─ 无对竞品的贬低性比较
```

## 🔍 核心流程（三阶段制）

> **设计原则**：每阶段有明确输入→输出→人工确认点。上阶段输出确认后才进入下阶段。

### ✅DO 阶段0：确认产品与信息来源

**目标**：搞清楚"填哪个产品、有哪些信息源可用"

**收到指令后，先确认**：

```
收到。先确认一下：
```

**审核人提供信息后，AI执行以下操作**：

1. 读取 brief工具HTML，确认字段结构（参考 `references/brief工具字段对照表.md`）
2. 🟢 读取文案风格参考案例（休足贴 + 蒸汽眼罩 + 三伏贴 三个完整brief），建立本产品的文案基调
3. 🟢 确定输出路径并复制模板、初始化JSON数据文件：

   > `/tmp/brief_data.json` 是JSON数据文件，记录已确认的字段值。每版块确认后更新此文件，再用 injection 脚本注入 HTML。
4. 读取所有产品信息源文件

- 读取产品信息源文件：成功=汇总覆盖范围 | 失败=标注该文件不可读并跳过，用剩余文件继续 | 重试=1次 | 降级=询问审核人是否有替代文件

→ 审核人确认 → 进入阶段1

### ✅DO 阶段1：逐版块填写 & 即时注入HTML

**目标**：按八大版块顺序，对照信息源逐字段填写，**每版块确认后立即更新JSON数据并注入HTML**。审核人需在浏览器中**刷新页面**查看最新填写效果（HTML为磁盘静态文件，浏览器不会自动感知文件变更）。

🧷 **上步校验锁**：阶段1执行前必须确认阶段0产出物（信息源覆盖范围清单 + HTML副本）已就绪且经审核人确认。若阶段0未完成，阶段1不得启动。

**填写规则**：

| 规则 | 标注 | 说明 |
|------|:---:|------|
| 有据可依 | ✅DO | 信息来源中有明确内容 → 直接引用，标注出处 |
| 无据留空 | ✅DO | 信息来源中无此内容 → 标注「信息源未提供，跳过」 |
| 待搜索 | ☑SHOULD | 可推断但需要验证 → 标注「待确认」，提议联网搜索 |
| 逻辑校验 | ✅DO | 同一信息在多个来源中出现 → 对比核验一致性 |
| 禁脑补 | ⛔DO NOT | 绝对禁止自行编造不存在于任何来源中的内容 |

---

**每版块的三步循环**（A→B→C，缺一不可）：

#### 步骤A：展示填写内容 → 审核人确认

```

| 字段 | 填写内容 | 信息来源 |
|------|---------|---------|
| 字段A | XXX | 产品信息卡-P2 |
| 字段B | ⚠️ 待确认 | 信息源未提供 |
| 字段C | — 跳过 | 选填且无信息源 |
| 字段D | 📝AI参考：[生成内容] | 参考案例风格，基于已确认的[X]生成 |

【待确认项汇总】
- 字段B：信息源未提供，是否联网搜索？
- 字段D：AI参考方案，是否接受或需要修改？

【⚖️ 合规自检】（涉及功效描述的版块）
- 无禁用词 √
- 功效已转为感受描述 √
- 日用品免责已包含 √

【struct 计数】（本版块涉及的动态列表）
- 成分: N 项 | 竞品: N 项 | 人群: N 项 | 场景: N 项 | POP: N 项 | POD: N 项 | QA: N 项

请确认以上内容。确认后我将更新JSON数据并注入HTML。
```

#### 步骤B：审核人确认后 → 更新JSON数据 → 注入HTML

AI 将确认的字段值记录到 `/tmp/brief_data.json`，然后用 `brief_inject.py` 脚本将整个数据注入 HTML。

**数据格式**（`/tmp/brief_data.json`）：
```json
{
  "fields": {
    "f-s-name": "蔡氏福宁·泡脚包",
    "f-s-category": "中药泡脚",
    "f-ing-1-name": "艾叶",
    "f-ing-1-func": "温经散寒",
    ...
  },
  "struct": {
    "ingCount": 2,
    "personaCount": 1,
    "sceneCount": 3,
    "popCount": 2,
    "podCount": 2,
    "compCount": 3,
    "qaCount": 2
  }
}
```

> **struct 计数规则**：记录动态列表（成分/竞品/人群/场景/POP/POD/QA）各有多少项。浏览器加载时根据这些计数重建对应的表单元素。

**写入三步操作**（每版块执行一次）：

```bash
# 第1步：更新 JSON 数据文件（追加/更新当前版块的字段值）
python3 -c "
import json
with open('/tmp/brief_data.json') as f:
    data = json.load(f)
# 更新字段值（示例——AI根据版块实际内容替换）
data['fields']['f-s-name'] = '蔡氏福宁·泡脚包'
data['fields']['f-s-category'] = '中药泡脚'
data['struct']['ingCount'] = 2
with open('/tmp/brief_data.json', 'w') as f:
    json.dump(data, f, ensure_ascii=False)
"

# 第2步：注入 HTML（将JSON数据以base64编码写入HTML中的预加载脚本）
python3 [Skill目录]/scripts/brief_inject.py [输出目录]/蔡氏福宁_产品brief_[产品名].html /tmp/brief_data.json

# 第3步：验证注入结果
python3 -c "
import json, base64, re
with open('[输出目录]/蔡氏福宁_产品brief_[产品名].html') as f:
    html = f.read()
m = re.search(r'atob\\(\"([^\"]+)\"\\)', html)
if m:
    data = json.loads(base64.b64decode(m.group(1)))
    print(f'OK: {sum(1 for v in data[\"fields\"].values() if v and str(v).strip())} fields loaded')
else:
    print('FAIL: preload script not found')
"
```

**注入脚本工作原理**（v1.5.0 双通道方案）：
- `brief_inject.py` 写入两条数据通道：
  - **通道1（主）**：`window.__BRIEF_DATA = {...}` — 直接在 HTML 中声明全局 JS 变量。**不依赖 localStorage**，在任何环境（WorkBuddy 内置预览 / 隐私模式 / 沙箱 iframe）都能加载
  - **通道2（备用）**：base64 → localStorage — 用于数据持久化保存（用户修改后自动保存到此通道）
- 用户打开 HTML 时，init 代码优先读取通道1（`window.__BRIEF_DATA`），写入 localStorage 备份；如果通道1不存在则从 localStorage 读取
- **关键改进**：通道1 完全绕过了"localStorage 不可用"的场景（WorkBuddy 预览面板 / 隐私浏览模式等），数据作为 JS 代码直接在页面解析时生效

> 动态列表容器ID对照：成分=`ingredient-list` | 竞品=`competitor-list` | 人群=`persona-list` | 场景=`scene-list` | POP=`pop-list` | POD=`pod-list` | QA=`qa-list`

---

> ⛔ **防沉默失败铁律**：brief_inject.py 输出格式为 `OK: N fields injected (ing:X persona:Y ...)`。若输出不以 `OK:` 开头或不含 `fields injected`，说明注入失败。AI **绝对不得**在没有看到 `OK:` 输出的情况下声称"写入成功"——必须以脚本的实际输出为准。

---

#### 步骤C：验证注入结果（AI回复"成功"前必须执行）

**判断标准**：
- `brief_inject.py` 输出 `OK:` 开头 → ✅ 写入成功，可告知用户
- 输出 `ERROR:` 或抛异常 → ❌ 写入失败，检查：HTML文件路径、JSON文件是否存在且格式正确 → 修正后重试

**验证通过后输出**：

```
✅ {版块名称} 已写入 HTML（brief_inject.py 输出: OK: N fields injected）
📄 文件：[工作区]/蔡氏福宁_产品brief_[产品名].html
🔄 刷新浏览器（F5/Cmd+R）查看最新内容，确认无误后进入下一版块。
💡 数据通过双通道注入（直接JS变量 + localStorage），所有浏览器环境均可用。进度条下方的诊断横幅会显示加载状态。
```

→ 审核人在浏览器中确认 → 进入下一个版块 → 重复步骤A→B→C

**八个版块全部确认并写入HTML后，进入阶段2。**

📦 **产出物**：逐版块增量更新的HTML文件（每版块确认后即注入，无需最后统一填写）

🔌 **熔断器**：
- JSON数据更新：成功=继续 | 失败=检查JSON格式 | 重试=2次 | 降级=手动编辑JSON文件
- brief_inject.py 注入：成功=输出 `OK: N fields` → 继续 | 失败=输出 `ERROR:` 或异常 → 检查文件路径和权限 | 重试=2次 | 降级=记录该版块未注入的字段，在阶段2统一修复
- Python验证注入：Python脚本输出 `OK:` → 写入成功，告知用户 | 输出 `FAIL:` → 写入失败，检查注入脚本后重试 | 重试=2次 | 降级=标注该字段写入失败，请审核人手动填写

## 🚫 禁止行为

| # | 禁止 | 原因 |
|---|------|------|
| 1 | 编造不存在于任何来源中的内容 | 零幻觉铁律——基于虚假内容的brief会误导后续决策 |
| 2 | 跳过审核人确认自动进入下一版块 | 每版块确认是防错根基 |
| 3 | 在有明确信息源时使用搜索内容替代 | 搜索内容权威性低于官方产品文件 |
| 4 | 填写信息源中有矛盾的内容而不标注 | 审核人必须知悉并做出选择 |
| 5 | 将"推测""可能"类内容当作确定内容填写 | 不确定就是不确定，标注待确认 |
| 6 | 在一次回复中输出多个版块的填写结果 | 每版块独立确认，避免信息过载 |
| 7 | 对选填字段强制要求信息来源 | 选填字段无信息源直接跳过即可 |
| 8 | 直接修改Skill目录下的原始HTML模板 | 模板是干净的源文件。必须先复制到工作区，只修改副本 |
| 9 | 版块确认后跳过JSON更新和brief_inject.py注入、等到全部填完再统一写 | 一次写全部容易漏填。逐版块注入才能在浏览器中及时核对 |
| 10 | brief_inject.py 返回后不检查输出就直接声称"写入成功" | 必须以脚本输出 `OK:` 开头为准。跳过输出检查是沉默失败的根源——HTML 实际空白，AI 却报告成功 |
| 11 | 用 perl 或其他方式直接修改 HTML DOM | brief工具数据源是 **localStorage**，修改 DOM 不会持久化。必须通过 JSON + brief_inject.py 方案写入 |
| 12 | 📝AI参考方案不加前缀、不标注来源 | 必须明确区分"产品信息原文"和"AI基于案例风格生成的参考方案"，否则审核人无法判断哪些是事实哪些是建议 |
| 13 | 在功效描述中使用禁用词（治疗/消炎/预防疾病/最/第一等） | 蔡氏福宁产品属于日用品，暗示医疗功效违反广告法。详见合规规范禁用词清单 |
| 14 | AI参考方案跳过合规审查直接填写 | AI生成内容同样必须过合规校验——生成的内容更容易触碰医疗暗示红线 |

## 💭 Alternatives Considered（设计决策记录）

### 决策1：为什么用三阶段制而非两阶段或单阶段？

| 方案 | 优点 | 为什么不选 |
|:-----|:-----|:----------|
| 单阶段（一口气读完文件→填完输出） | 速度快 | ⛔ 无法中途纠错——信息源覆盖不全时已生成了大量不可靠内容。违反「每版块确认」铁律 |
| 两阶段（确认信息源→直接填完输出） | 比三阶段少一次交互 | ⛔ 版块间无确认门禁——前版块错误会级联污染后版块 |
| **三阶段（确认信息源→逐版块确认→输出）✅** | 每版块独立确认，错误不级联 | 成本：比两阶段多N次交互（N=版块数），但每个版块平均3-5条记录，确认负担可接受 |

### 决策2：为什么逐版块即时写入HTML而非最后统一填写？（v1.1.0 重写，v1.3.0 修订）

| 方案 | 优点 | 为什么不选 |
|:-----|:-----|:----------|
| 最后统一填写HTML（v1.0.x 方案） | 实现简单，一次操作 | ⛔ 8个版块配齐后一次性核对，信息量大（~50+字段），审核人容易漏读缺失项 |
| **逐版块即时写入HTML ✅** | 每版块3-7个字段，审核人可在浏览器中刷新后逐个核对，漏检概率大幅降低 | AI需要在每版块后额外执行注入脚本，但单次注入仅需1-2秒。⚠️ 审核人每次需手动刷新浏览器（F5/Cmd+R）才能看到最新写入内容 |


### 决策3：为什么不使用"一次输出所有版块"模式？

一次输出所有版块会导致：信息过载（8版块×平均4-6字段=逾50字段在一次回复中输出）→ 审核人注意力分散 → 漏检概率增加。逐版块输出确保每次审核人只面对3-7条待确认信息。

### 决策4：为什么模板复制放在阶段0而非阶段2？（v1.1.0 重写）

| 方案 | 优点 | 为什么不选 |
|:-----|:-----|:----------|
| 在阶段2统一复制 | 阶段0/1无需关心文件系统 | ⛔ 阶段1无法逐版块写入HTML——没有写入目标 |
| **在阶段0提前复制 ✅** | 阶段1的每版块确认后立即有HTML文件可写入、可查看 | 阶段0多一步文件操作，但这步是可逆的（副本写坏了重新复制即可） |

### 决策5：双格式Markdown解析器的价值（保留）

v1.0.2 新增的 `parseSimpleMarkdown()` 双解析器仍然保留在HTML工具中。虽然主流程改为AI直接写HTML，但HTML工具的「导入Markdown」功能仍然可用——当审核人需要从历史MD文件恢复数据、或手动准备MD内容时，双解析器确保两种格式都能正确导入。

### 决策6：为什么用 JSON + localStorage 脚本注入替代 perl DOM 写入？（v1.3.0 新增）

| 方案 | 优点 | 为什么不选 |
|:-----|:-----|:----------|
| perl 正则修改 DOM（v1.0.x ~ v1.2.x） | 思路直接：改HTML源码 | ⛔ **致命缺陷**：brief工具的数据源是 **localStorage**，不是 HTML DOM。静态字段的 `<input value="xxx">` 在页面刷新时存在竞态条件；动态字段（成分/竞品/人群等7类）完全由 JS 在运行时生成，HTML 源码中容器为空——perl 永远匹配不到目标。写入成功但空白的原因是数据根本没进入 localStorage |
| **JSON + base64 脚本注入 ✅** | 同时解决静态和动态字段。base64 编码消除所有转义问题。数据在页面加载时通过 `<script>` 预填 localStorage，`loadFromLocal()` 读取后正常渲染 | 需要 Python 3 环境（WorkBuddy 内置） |
