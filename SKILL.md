# 蔡氏福宁产品brief填写 v1.0.1

> **版本**：v1.0.1 | **日期**：2026-06-30

## 概述

帮助审核人将产品信息文件中的内容，结构化填写到「蔡氏福宁 · 产品brief 填写工作台」中。

**brief工具路径**：与本 SKILL.md 同目录下的 `蔡氏福宁_产品brief_填写工具.html`

**核心原则**：有据可依、无据留空、宁缺毋滥、逻辑自洽。
**防幻觉铁律**：凡不存在于信息来源中的内容，不得自行编造。

---

## ⚠️ 铁律声明

**CRITICAL — 本Skill有两条核心铁律，优先级高于一切：**

⛔DO NOT 1. **零幻觉，严禁脑补** — 每一条填入的内容必须有明确的信息来源。信息来源中不存在的内容，不得自行编造。不确定的标注「待确认」，由审核人决定。
✅DO 2. **每版块确认后方可进入下一版块** — 一个版块的填写结果必须经审核人确认，AI不得自动跳版块。

## 触发条件

### 应该触发
- 用户说：产品brief填写、产品信息检查、产品信息核对、帮我把产品信息输入到brief工具里
- 场景：将产品信息文件（PDF/DOCX/XLSX/图片/企微文档）中的内容填充到brief工具中
- 场景：已有产品brief内容需要人工核对信息准确性

### 不应该触发
- 不涉及产品brief填写的纯咨询
- 文案合规审核（应使用「蔡氏福宁文案审核」Skill）
- 从零开始的市场调研
- 非蔡氏福宁品牌的产品填写

## 🔍 核心流程（三阶段制）

> **设计原则**：每阶段有明确输入→输出→人工确认点。上阶段输出确认后才进入下阶段。

---

### ✅DO 阶段0：确认产品与信息来源

**目标**：搞清楚"填哪个产品、有哪些信息源可用"

**收到指令后，先确认**：

```
收到。先确认一下：

1. 要填写哪个产品的brief？
2. 产品信息文件有哪些？（直接发文件给我，或告诉我路径）
3. 有没有特别想重点确认的版块？
```

**审核人提供信息后，AI执行以下操作**：

1. 读取 brief工具HTML，确认字段结构（参考 `references/brief工具字段对照表.md`）
2. 读取所有产品信息源文件
3. 汇总可用信息源的覆盖范围，告知审核人：

```
📋 信息来源确认：

已读取以下文件：
- [文件名1] → 覆盖：产品名称、规格、成分、使用规范
- [文件名2] → 覆盖：品牌调性、口号方向
- ...

⚠️ 以下版块在现有信息源中暂无数据：
- [版块名] — 如需要可联网搜索补充

确认后，我按版块逐个填写。
```

📦 **产出物**：信息源覆盖范围清单（含已读取文件列表 + 已覆盖/未覆盖版块标注）

🔌 **熔断器**：
- 读取brief工具HTML：成功=继续 | 失败=提示用户检查brief工具路径 | 重试=1次 | 降级=使用 `references/brief工具字段对照表.md` 作为备用字段结构
- 读取产品信息源文件：成功=汇总覆盖范围 | 失败=标注该文件不可读并跳过，用剩余文件继续 | 重试=1次 | 降级=询问审核人是否有替代文件

→ 审核人确认 → 进入阶段1

---

### ✅DO 阶段1：逐版块对照填写

**目标**：按八大版块顺序，对照信息源逐字段填写。每版块填写完毕交审核人确认。

🧷 **上步校验锁**：阶段1执行前必须确认阶段0产出物（信息源覆盖范围清单）已存在且经审核人确认。若阶段0未完成，阶段1不得启动。

**填写规则**：

| 规则 | 标注 | 说明 |
|------|:---:|------|
| 有据可依 | ✅DO | 信息来源中有明确内容 → 直接引用，标注出处 |
| 无据留空 | ✅DO | 信息来源中无此内容 → 标注「信息源未提供，跳过」 |
| 待搜索 | ☑SHOULD | 可推断但需要验证 → 标注「待确认」，提议联网搜索 |
| 逻辑校验 | ✅DO | 同一信息在多个来源中出现 → 对比核验一致性 |
| 禁脑补 | ⛔DO NOT | 绝对禁止自行编造不存在于任何来源中的内容 |

📦 **产出物**：逐版块填写表（含字段值 + 信息来源标注 + 待确认项汇总）

**每版块输出格式**：

```
📦 [版块名称]

| 字段 | 填写内容 | 信息来源 |
|------|---------|---------|
| 产品名称 | 蔡氏福宁·XXX | 产品信息卡-P2 |
| 品类定义 | [填写内容] | 品牌手册-P5 |
| ... | ... | ... |
| XX字段 | ⚠️ 待确认 | 信息源未提供 |
| XX字段 | — 跳过 | 选填且无信息源 |

【待确认项汇总】
- XX字段：信息源未提供，是否联网搜索？
- YY字段：多个来源说法不一致（A说XX / B说YY），请确认以哪个为准？

以上内容是否准确？确认后进入下一版块。
```

**每版块确认后，进入下一版块。八个版块全部确认完毕后，进入阶段2。**

---

### ✅DO 阶段2：填写到brief工具 & 输出

**目标**：将确认后的内容填入brief工具HTML，输出可用的brief文件。

**操作方式**：AI通过修改HTML文件的 `value` / `innerText`，将阶段1确认的内容填入对应字段。

> 详细操作见 `references/brief填写操作指南.md`

📦 **产出物**：
1. 填写完成的HTML文件（原地覆写或另存为 `蔡氏福宁_产品brief_[产品名].html`）
2. Markdown格式的brief摘要

🔌 **熔断器**：
- 写入HTML文件：成功=继续 | 失败=提示权限/路径错误 | 重试=1次 | 降级=输出Markdown摘要作为替代产物
- 导出Markdown摘要：成功=完成 | 失败=提示手动复制 | 降级=提示审核人手动从HTML复制

**阶段2完成条件**：审核人确认brief内容无误。

🔒 **完整性锁**（阶段2输出前必须全部通过）：

- [ ] 全部八个版块已获审核人逐版块确认
- [ ] 每个字段有可追溯的信息来源标注（"信息源未提供"/"联网搜索"/文件路径）
- [ ] 无"⚠️ 待确认"项被当作确定值填入HTML
- [ ] 交叉验证通过——有差异的字段已标注差异并获审核人确认
- [ ] HTML文件路径有效、写入成功

## 🧠 思考检查点

全部三个阶段执行完毕后，AI必须逐项自检：

```
🧠 思考检查点
├─ 数据完整：八个版块全部填写？无遗漏版块？
├─ 来源可溯：每个字段值标注了信息来源（文件路径/"联网搜索"/"信息源未提供"）？
├─ 无脑补：所有填写内容均出自信息源，无一行为自行编造？
├─ 差异已标注：交叉验证中有矛盾的内容已标注并获确认？
├─ 待确认已呈现：所有"⚠️ 待确认"项已汇总并呈现给审核人？
├─ HTML已写入：brief工具HTML文件已更新、可正常打开？
└─ 确认闭环：审核人已确认最终结果？
```

## 🧠 交叉验证规则

当同一信息在多个来源中出现时，AI必须执行交叉验证：

1. **完全一致** → 直接使用，标注"多源一致"
2. **有差异** → 标注差异点，请审核人确认以哪个为准
3. **仅一处有** → 使用该来源，标注"单一来源"

**常见需交叉验证的内容**：
- 产品名称（产品信息卡 vs 包装稿 vs VI文件）
- 规格/片数（产品信息卡 vs 渠道物料表）
- 成分列表（产品信息卡 vs 包装背面 vs 企标）
- 保质期（产品信息卡 vs 包装背面）

## 🔍 联网搜索规则

当信息源未覆盖某字段时，AI可以提议联网搜索：

| 可搜索内容 | 不可搜索内容 |
|-----------|------------|
| 竞品信息（补充市场层） | 蔡氏福宁内部产品数据 |
| 成分公知功效（补充产品层） | 未公开的产品配方 |
| 消费者常见问题（补充QA） | 内部定价策略 |
| 品类市场趋势（补充战略层） | 未公开的渠道数据 |

🔌 **熔断器**：
- 联网搜索：成功=将结果标注为"联网搜索来源"填入 | 失败=标注"搜索不可用，跳过" | 重试=1次 | 降级=标注「待确认」，建议审核人手动搜索

**✅DO 搜索前必须征得审核人同意**：
```
以下字段信息源未覆盖，是否开启联网搜索补充？
- [字段名]：可搜索 [搜索方向]
- [字段名]：可搜索 [搜索方向]
```

## 🚫 禁止行为

| # | 禁止 | 原因 |
|---|------|------|
| 1 | 编造不存在于任何来源中的内容 | 零幻觉铁律——基于虚假内容的brief会误导后续决策 |
| 2 | 跳过审核人确认自动进入下一版块 | 每版块确认是防错根基 |
| 3 | 在有明确信息源时使用搜索内容替代 | 搜索内容权威性低于官方产品文件 |
| 4 | 填写信息源中有矛盾的内容而不标注 | 审核人必须知悉并做出选择 |
| 5 | 将"推测""可能"类内容当作确定内容填写 | 不确定就是不确定，标注待确认 |
| 6 | 在一次回复中输出多个版块的填写结果 | 每版块独立确认，避免信息过载 |
| 7 | 对选填字段强制要求信息来源 | 选填字段无信息源直接跳过即可 |

## 💭 Alternatives Considered（设计决策记录）

### 决策1：为什么用三阶段制而非两阶段或单阶段？

| 方案 | 优点 | 为什么不选 |
|:-----|:-----|:----------|
| 单阶段（一口气读完文件→填完输出） | 速度快 | ⛔ 无法中途纠错——信息源覆盖不全时已生成了大量不可靠内容。违反「每版块确认」铁律 |
| 两阶段（确认信息源→直接填完输出） | 比三阶段少一次交互 | ⛔ 版块间无确认门禁——前版块错误会级联污染后版块 |
| **三阶段（确认信息源→逐版块确认→输出）✅** | 每版块独立确认，错误不级联 | 成本：比两阶段多N次交互（N=版块数），但每个版块平均3-5条记录，确认负担可接受 |

### 决策2：为什么方案A（Markdown导入）优先于方案B（DOM编辑）？

| 方案 | 优点 | 为什么不优先 |
|:-----|:-----|:------------|
| 方案B：AI直接编辑HTML DOM | 全自动化 | ⛔ 动态DOM元素初始无预设值，需分析JS函数签名→修改JS逻辑→高风险。HTML工具为该Skill的同目录文件，意外损坏后不可恢复 |
| **方案A：Markdown导入 ✅** | 无DOM操作风险，审核人可预览和微调 | 需要审核人手动粘贴一次，但比修复损坏的HTML成本低得多 |

### 决策3：为什么不使用"一次输出所有版块"模式？

一次输出所有版块会导致：信息过载（8版块×平均4-6字段=逾50字段在一次回复中输出）→ 审核人注意力分散 → 漏检概率增加。逐版块输出确保每次审核人只面对3-7条待确认信息。
