# 蔡氏福宁产品brief填写 v1.1.0

> **版本**：v1.1.0 | **日期**：2026-07-03

## 概述

帮助审核人将产品信息文件中的内容，结构化填写到「蔡氏福宁 · 产品brief 填写工作台」中。

**brief工具路径**：与本 SKILL.md 同目录下的 `蔡氏福宁_产品brief_填写工具.html`

**核心原则**：有据可依、无据留空、宁缺毋滥、逻辑自洽。
**防幻觉铁律**：凡不存在于信息来源中的内容，不得自行编造。

---

## ⚠️ 铁律声明

**CRITICAL — 本Skill有两条核心铁律，优先级高于一切：**

⛔DO NOT 1. **零幻觉，严禁脑补** — 每一条填入的内容必须有明确的信息来源。信息来源中不存在的内容，不得自行编造。不确定的标注「待确认」，由审核人决定。
✅DO 2. **每版块确认后方可进入下一版块** — 一个版块的填写结果必须经审核人确认，AI不得自动跳版块。

## 触发条件

### 应该触发
- 用户说：产品brief填写、产品信息检查、产品信息核对、帮我把产品信息输入到brief工具里
- 场景：将产品信息文件（PDF/DOCX/XLSX/图片/企微文档）中的内容填充到brief工具中
- 场景：已有产品brief内容需要人工核对信息准确性

### 不应该触发
- 不涉及产品brief填写的纯咨询
- 文案合规审核（应使用「蔡氏福宁文案审核」Skill）
- 从零开始的市场调研
- 非蔡氏福宁品牌的产品填写

## 🔍 核心流程（三阶段制）

> **设计原则**：每阶段有明确输入→输出→人工确认点。上阶段输出确认后才进入下阶段。

---

### ✅DO 阶段0：确认产品与信息来源

**目标**：搞清楚"填哪个产品、有哪些信息源可用"

**收到指令后，先确认**：

```
收到。先确认一下：

1. 要填写哪个产品的brief？
2. 产品信息文件有哪些？（直接发文件给我，或告诉我路径）
3. brief 输出到哪？默认：当前 WorkBuddy 工作区/[产品名]/
   如需指定其他目录请告诉我。
4. 有没有特别想重点确认的版块？
```

**审核人提供信息后，AI执行以下操作**：

1. 读取 brief工具HTML，确认字段结构（参考 `references/brief工具字段对照表.md`）
2. 🟢 确定输出路径并复制模板（原始模板永不修改）：
   - 默认路径：`[当前WorkBuddy工作区]/[产品名]/蔡氏福宁_产品brief_[产品名].html`
   - 审核人指定了其他目录则使用指定目录
   - 确保输出目录存在（不存在则自动创建）
   ```
   mkdir -p [输出目录]
   cp [Skill目录]/蔡氏福宁_产品brief_填写工具.html [输出目录]/蔡氏福宁_产品brief_[产品名].html
   ```
3. 读取所有产品信息源文件
4. 汇总可用信息源的覆盖范围，告知审核人：

```
📋 信息来源确认：

已读取以下文件：
- [文件名1] → 覆盖：产品名称、规格、成分、使用规范
- [文件名2] → 覆盖：品牌调性、口号方向
- ...

⚠️ 以下版块在现有信息源中暂无数据：
- [版块名] — 如需要可联网搜索补充

确认后，我按版块逐个填写。
```

📦 **产出物**：信息源覆盖范围清单 + 输出目录中的HTML副本（默认 `[WorkBuddy工作区]/[产品名]/` 下，从模板复制而来）

🔌 **熔断器**：
- 读取brief工具HTML：成功=继续 | 失败=提示用户检查brief工具路径 | 重试=1次 | 降级=使用 `references/brief工具字段对照表.md` 作为备用字段结构
- 复制模板到工作区：成功=继续 | 失败（如路径不存在）=提示审核人确认工作区路径并重试 | 重试=1次 | 降级=停止流程，等待正确的路径
- 读取产品信息源文件：成功=汇总覆盖范围 | 失败=标注该文件不可读并跳过，用剩余文件继续 | 重试=1次 | 降级=询问审核人是否有替代文件

→ 审核人确认 → 进入阶段1

---

### ✅DO 阶段1：逐版块填写 & 即时写入HTML

**目标**：按八大版块顺序，对照信息源逐字段填写，**每版块确认后立即写入HTML**，审核人可在浏览器中实时查看填写效果。

🧷 **上步校验锁**：阶段1执行前必须确认阶段0产出物（信息源覆盖范围清单 + HTML副本）已就绪且经审核人确认。若阶段0未完成，阶段1不得启动。

**填写规则**：

| 规则 | 标注 | 说明 |
|------|:---:|------|
| 有据可依 | ✅DO | 信息来源中有明确内容 → 直接引用，标注出处 |
| 无据留空 | ✅DO | 信息来源中无此内容 → 标注「信息源未提供，跳过」 |
| 待搜索 | ☑SHOULD | 可推断但需要验证 → 标注「待确认」，提议联网搜索 |
| 逻辑校验 | ✅DO | 同一信息在多个来源中出现 → 对比核验一致性 |
| 禁脑补 | ⛔DO NOT | 绝对禁止自行编造不存在于任何来源中的内容 |

---

**每版块的两步循环**：

#### 步骤A：展示填写内容 → 审核人确认

```
📦 {N}/8 {版块名称}

| 字段 | 填写内容 | 信息来源 |
|------|---------|---------|
| 字段A | XXX | 产品信息卡-P2 |
| 字段B | ⚠️ 待确认 | 信息源未提供 |
| 字段C | — 跳过 | 选填且无信息源 |

【待确认项汇总】
- 字段B：信息源未提供，是否联网搜索？

请确认以上内容。确认后我将写入HTML。
```

#### 步骤B：审核人确认后 → 写入HTML → 展示文件路径

AI使用 `perl` 将确认的字段值写入HTML副本。写入规则：

| 字段类型 | 写入方式 |
|---------|---------|
| `<input>` 字段 | 添加 `value="内容"` 属性 |
| `<textarea>` 字段 | 替换标签间文本内容 |
| 动态列表（成分/竞品/人群等） | 在对应容器 `<div id="{list-name}">` 结束标签前插入HTML结构 |

**写入命令模板**：

> ⚠️ 字段内容中若含 `$` `@` `\` `&` `/` `"` 等 perl 特殊字符，必须先转义再代入命令。推荐方法：将内容写入临时文件，perl 读取文件内容填入，避免 shell 层面的二次转义。

```bash
# input 字段（单行）—— 内容无特殊字符时直接用
perl -i -pe 's/(<input[^>]*id="FIELD_ID"[^>]*)(>)/\1 value="填写内容"\2/' HTML_FILE

# textarea / input 字段 —— 推荐方式：内容写入临时文件后读取，避免转义问题
echo '填写内容' > /tmp/brief_tmp.txt
perl -i -0777 -pe '
  open(F, "/tmp/brief_tmp.txt"); $v = do { local $/; <F> }; close(F); chomp $v;
  s{(<textarea[^>]*id="FIELD_ID"[^>]*>)(.*?)(</textarea>)}{\1$v\3}gs
' HTML_FILE

# input 字段 —— 同样用临时文件方式（内容含特殊字符时）
perl -i -pe '
  open(F, "/tmp/brief_tmp.txt"); $v = do { local $/; <F> }; close(F); chomp $v;
  s/(<input[^>]*id="FIELD_ID"[^>]*)(>)/\1 value="$v"\2/
' HTML_FILE

# 动态列表 —— 追加到指定容器结束标签前（精确容器ID匹配）
echo '动态HTML结构' > /tmp/brief_tmp.txt
perl -i -0777 -pe '
  open(F, "/tmp/brief_tmp.txt"); $ins = do { local $/; <F> }; close(F);
  s{(<div id="CONTAINER_ID">[\s\S]*?)(</div>)}{\1$ins\2}m
' HTML_FILE
```

> 动态列表容器ID对照：成分=`ingredient-list` | 竞品=`competitor-list` | 人群=`persona-list` | 场景=`scene-list` | POP=`pop-list` | POD=`pod-list` | QA=`qa-list`

写入完成后输出：

```
✅ {版块名称} 已写入 HTML：[工作区]/蔡氏福宁_产品brief_[产品名].html
🔍 请打开该文件在浏览器中查看，确认无误后进入下一版块。
```

→ 审核人在浏览器中确认 → 进入下一个版块 → 重复步骤A→B

**八个版块全部确认并写入HTML后，进入阶段2。**

📦 **产出物**：逐版块增量更新的HTML文件（每版块确认后即写入，无需最后统一填写）

🔌 **熔断器**：
- 写入HTML字段值：成功=继续 | 失败=提示字段ID不匹配 | 重试=1次（检查字段ID是否正确） | 降级=记录该版块未写入的字段，在阶段2统一修复

---

### ✅DO 阶段2：最终交付

**目标**：八个版块HTML已全部填写完成，输出最终交付物并向审核人说明使用方式。

✅DO 八个版块已在阶段1逐版块写入HTML，阶段2无需再次写入。

**最终提示模板**（必须输出）：

```
📦 产品brief 填写完成

| 文件 | 路径 |
|------|------|
| 🌐 HTML工作台 | [工作区]/蔡氏福宁_产品brief_[产品名].html |

📥 使用方式：
1. 双击打开 HTML 文件，在浏览器中查看完整brief
2. 可点击「📝 导出Markdown」生成 .md 备份文件
3. 可点击「📄 下载PDF」导出打印版
4. 如需调整，直接在浏览器中修改后自动保存（localStorage）

- 核心价值：[POD数量] 个差异点

需要我帮你做其他调整吗？
```

📦 **产出物**：填写完成的HTML文件（已包含全部八个版块的内容，可直接使用）

🔒 **完整性锁**（阶段2输出前必须全部通过）：

- [ ] 全部八个版块已获审核人逐版块确认
- [ ] 每个版块的内容已成功写入HTML文件
- [ ] 每个字段有可追溯的信息来源标注
- [ ] 无"⚠️ 待确认"项被当作确定值填入HTML
- [ ] 交叉验证通过——有差异的字段已标注差异并获审核人确认
- [ ] HTML文件可正常在浏览器中打开并显示完整内容

## 🧠 交叉验证规则

当同一信息在多个来源中出现时，AI必须执行交叉验证：

1. **完全一致** → 直接使用，标注"多源一致"
2. **有差异** → 标注差异点，请审核人确认以哪个为准
3. **仅一处有** → 使用该来源，标注"单一来源"

**常见需交叉验证的内容**：
- 产品名称（产品信息卡 vs 包装稿 vs VI文件）
- 规格/片数（产品信息卡 vs 渠道物料表）
- 成分列表（产品信息卡 vs 包装背面 vs 企标）
- 保质期（产品信息卡 vs 包装背面）

- 联网搜索：成功=将结果标注为"联网搜索来源"填入 | 失败=标注"搜索不可用，跳过" | 重试=1次 | 降级=标注「待确认」，建议审核人手动搜索

**✅DO 搜索前必须征得审核人同意**：
```
以下字段信息源未覆盖，是否开启联网搜索补充？

- [字段名]：可搜索 [搜索方向]
```

## 🚫 禁止行为

| # | 禁止 | 原因 |
|---|------|------|
| 1 | 编造不存在于任何来源中的内容 | 零幻觉铁律——基于虚假内容的brief会误导后续决策 |
| 2 | 跳过审核人确认自动进入下一版块 | 每版块确认是防错根基 |
| 3 | 在有明确信息源时使用搜索内容替代 | 搜索内容权威性低于官方产品文件 |
| 4 | 填写信息源中有矛盾的内容而不标注 | 审核人必须知悉并做出选择 |
| 5 | 将"推测""可能"类内容当作确定内容填写 | 不确定就是不确定，标注待确认 |
| 6 | 在一次回复中输出多个版块的填写结果 | 每版块独立确认，避免信息过载 |
| 7 | 对选填字段强制要求信息来源 | 选填字段无信息源直接跳过即可 |
| 8 | 直接修改Skill目录下的原始HTML模板 | 模板是干净的源文件。必须先复制到工作区，只修改副本 |
| 9 | 版块确认后跳过HTML写入、等到全部填完再统一写 | 一次写全部容易漏填。逐版块写入才能在浏览器中及时核对 |

## 💭 Alternatives Considered（设计决策记录）

### 决策1：为什么用三阶段制而非两阶段或单阶段？

| 方案 | 优点 | 为什么不选 |
|:-----|:-----|:----------|
| 单阶段（一口气读完文件→填完输出） | 速度快 | ⛔ 无法中途纠错——信息源覆盖不全时已生成了大量不可靠内容。违反「每版块确认」铁律 |
| 两阶段（确认信息源→直接填完输出） | 比三阶段少一次交互 | ⛔ 版块间无确认门禁——前版块错误会级联污染后版块 |
| **三阶段（确认信息源→逐版块确认→输出）✅** | 每版块独立确认，错误不级联 | 成本：比两阶段多N次交互（N=版块数），但每个版块平均3-5条记录，确认负担可接受 |

### 决策2：为什么逐版块即时写入HTML而非最后统一填写？（v1.1.0 重写）

| 方案 | 优点 | 为什么不选 |
|:-----|:-----|:----------|
| 最后统一填写HTML（v1.0.x 方案） | 实现简单，一次操作 | ⛔ 8个版块配齐后一次性核对，信息量大（~50+字段），审核人容易漏读缺失项 |
| **逐版块即时写入HTML ✅** | 每版块3-7个字段，审核人可立即在浏览器中核对，漏检概率大幅降低 | AI需要在每版块后额外执行perl命令写入，但单次写入仅需1-2秒 |


### 决策3：为什么不使用"一次输出所有版块"模式？

一次输出所有版块会导致：信息过载（8版块×平均4-6字段=逾50字段在一次回复中输出）→ 审核人注意力分散 → 漏检概率增加。逐版块输出确保每次审核人只面对3-7条待确认信息。

### 决策4：为什么模板复制放在阶段0而非阶段2？（v1.1.0 重写）

| 方案 | 优点 | 为什么不选 |
|:-----|:-----|:----------|
| 在阶段2统一复制 | 阶段0/1无需关心文件系统 | ⛔ 阶段1无法逐版块写入HTML——没有写入目标 |
| **在阶段0提前复制 ✅** | 阶段1的每版块确认后立即有HTML文件可写入、可查看 | 阶段0多一步文件操作，但这步是可逆的（副本写坏了重新复制即可） |

### 决策5：双格式Markdown解析器的价值（保留）

v1.0.2 新增的 `parseSimpleMarkdown()` 双解析器仍然保留在HTML工具中。虽然主流程改为AI直接写HTML，但HTML工具的「导入Markdown」功能仍然可用——当审核人需要从历史MD文件恢复数据、或手动准备MD内容时，双解析器确保两种格式都能正确导入。
