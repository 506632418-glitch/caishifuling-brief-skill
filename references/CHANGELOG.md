## v1.5.1 (2026-07-05) — 刷新按钮修复：缓存穿透

- 🐛 **修复**：「🔄 刷新数据」按钮使用 `location.reload()` 在内嵌 WebView（WorkBuddy 预览）中可能被缓存拦截，点击无反应
  - **根因**：`location.reload()` 可能从浏览器缓存加载而非重新读取磁盘文件
  - **修复**：改为 `location.href = url + '?_=' + Date.now()` —— 给 URL 加上变化的查询参数，强制浏览器绕过缓存重新请求

## v1.5.0 (2026-07-05) — 双通道数据注入（根治"注入成功但HTML空白"）

- 🔄 **架构级升级**：数据注入从单通道（localStorage）改为双通道
  - **通道1（主）**：`window.__BRIEF_DATA = {...}` — 直接在 HTML 中声明全局 JS 变量。完全绕过 localStorage
  - **通道2（备份）**：base64 → localStorage — 保留用于持久化
  - **根因**：之前所有数据都依赖 localStorage 加载。当浏览器环境不支持 localStorage（WorkBuddy 内嵌预览、隐私模式、沙箱 iframe）时，数据静默丢失。通道1 将数据作为 JS 代码直接嵌入 HTML，在任何环境中都能执行
- 🔧 `brief_inject.py`：preload 脚本重写为双通道
- 🔧 HTML init 代码：新增 `loadFromDirectData()` 备用加载器 + 双通道优先读取逻辑
- 🔧 SKILL.md 步骤C：移除"不要用内置预览"限制（双通道方案已解决此问题）
- 🔧 故障排查章节：简化排查清单，更新快速验证命令
- 📝 版本号：v1.4.1 → v1.5.0（中版本：架构级改动）

## v1.3.3 (2026-07-04) — 修复preload静默失败 + 清理逻辑竞态

- 🐛 **关键修复 #1**：`decodeURIComponent` hack 在特定字节序列上抛 `URIError`，导致 preload 脚本静默失败
  - **根因**：`decodeURIComponent` 对无效 UTF-8 字节序列抛异常（而非降级处理）。`atob` → binary string → `charCodeAt` → `%XX` → `decodeURIComponent` 这条链路中，某些合法 JSON 内容的字节组合可能触发此异常
  - **修复**：替换为 `TextDecoder("utf-8")` —— `atob()` → `Uint8Array` → `TextDecoder.decode()` → `JSON.parse`。`TextDecoder` 对无效字节序列降级为 U+FFFD 而非抛异常
  - 同时给 preload 脚本加 `try/catch` + `console.error`，失败不再静默
- 🐛 **关键修复 #2**：HTML 初始化清理逻辑与 preload 脚本存在竞态条件
  - **根因**：`DOMContentLoaded` 回调中 `removeItem('caishifuling_brief_v6')` 在未设置 `cleanup_done` 标记时执行——恰好抹掉 preload 脚本刚写入的数据（preload 在 DOMContentLoaded 之前运行）
  - **修复**：清理逻辑只 wipe v5，不再碰 v6
- 📝 SKILL.md 注入工作原理补充刷新说明

## v1.3.1 (2026-07-03) — 修复注入脚本中文乱码

- 🐛 **关键修复**：`brief_inject.py` 注入的 `<script id="brief-preload">` 中 `atob()` 不解码 UTF-8，导致中文全部乱码
  - **根因**：`atob()` 返回 "binary string"（每字符1字节），但中文 UTF-8 是 3 字节/字符。`JSON.parse(atob(b64))` 将多字节序列当成 Latin-1 逐字符处理，产生乱码
  - **修复**：加入 `decodeURIComponent` hack —— 将 binary string 中每个字节转为 `%XX` 格式，再用 `decodeURIComponent` 还原为正确的 UTF-8 字符串，最后才 `JSON.parse`
- 🔧 `scripts/brief_inject.py` 第 71-85 行：注入脚本模板修改，新增注释说明乱码原因
